# 🎯 나만의 퀴즈 게임 (My Quiz Game)

초보자도 쉽게 즐기고 배울 수 있는 **터미널 기반 파이썬 퀴즈 게임**입니다. 
프로그래밍 언어를 배우는 가장 확실한 방법은 직접 프로그램을 만들어보는 것입니다! 이 프로젝트는 파이썬(Python)의 기본 문법과 객체 지향(클래스), 그리고 파일 입출력을 활용하여 만들어졌으며, Git을 통한 버전 관리의 흔적을 고스란히 담고 있습니다.

---

## 📖 1. 프로젝트 개요
터미널 환경에서 실행되는 퀴즈 게임 프로그램입니다. 
파이썬 언어의 핵심 개념(조건문, 반복문, 클래스와 객체, JSON 파일 처리)을 바탕으로 설계되었으며, 올바르지 않은 사용자 입력(예: 문자 입력, 빈칸 등)에도 프로그램이 튕기거나 비정상 종료되지 않도록 견고하게 예외 처리(`try/except`)가 되어있습니다. 프로그램 종료 중에도 안전한 상태 저장을 보장합니다.

---

## 🎬 2. 퀴즈 주제 선정 이유
본 퀴즈 게임의 기본 주제는 **'영화 (Movie)'** 입니다.
마블 시네마틱 유니버스(타노스), 기생충, 인터스텔라 등 대중적으로 유명한 영화 지식들을 문제로 구성하였습니다. 누구나 가볍고 즐겁게 프로그래밍의 결과를 테스트해 볼 수 있도록 가장 친숙한 주제를 선정했습니다.

---

## 💻 3. 실행 방법
이 프로그램은 외부 라이브러리 설치 없이 순수 파이썬의 기본 기능만을 사용합니다.

1. 컴퓨터에 파이썬(Python 3.10 이상 권장)이 올바르게 설치되어 있는지 확인합니다.
2. 터미널(명령 프롬프트)을 열고, 본 프로젝트 폴더 안으로 이동합니다.
3. 아래의 명령어를 입력하여 게임을 실행합니다.
   ```bash
   python main.py
   ```

---

## 📋 4. 주요 기능 목록
프로그램을 실행하면 메뉴 화면이 나타납니다. 사용자는 1~6번의 번호를 입력해 원하는 기능을 자유롭게 수행할 수 있습니다.

1. **퀴즈 풀기**: 몇 문제를 풀지 선택한 뒤, 보유한 퀴즈를 무작위(Random) 순서로 풉니다. 
   - 문항당 기본 획득 점수는 **10점**입니다. 
   - 정답 입력 시 `0`번을 누르면 **힌트**를 볼 수 있으며, 힌트 사용 시 획득 점수는 **5점**으로 감점 채점됩니다.
2. **퀴즈 추가**: 프로그램 내에서 자신만의 새로운 퀴즈를 직접 등록할 수 있습니다. (문제 내용, 객관식 보기 4개, 정답 번호, 그리고 💡힌트까지 꼼꼼히 입력 가능합니다)
3. **퀴즈 목록**: 현재 저장소 파일에 등록된 모든 퀴즈들의 목록을 한눈에 조회합니다.
4. **퀴즈 삭제**: 삭제하려는 퀴즈의 번호를 선택하여 불필요한 문제를 목록에서 영구적으로 지울 수 있습니다.
5. **점수 및 히스토리 확인**: 본인의 **역대 최고 점수**와 함께, 가장 최근에 플레이한 5번의 **게임 기록**(플레이 날짜/시간, 문제 수, 획득 점수)을 깔끔하게 확인합니다.
6. **종료**: 현재까지 플레이한 상태를 안전하게 파일명(`state.json`)에 저장한 뒤 게임을 종료합니다. 

> ⚠️ 잘못된 숫자(예: 9번 입력), 문자의 오입력, 빈칸(`Enter`), `Ctrl+C` 등 어떠한 비정상 입력 상황이 발생해도 프로그램은 사용자 안내 메시지를 구동하며 안전하게 방어해냅니다.

## 📂 5. 파일 구조 설명
프로그램을 구성하는 핵심 파일들은 역할에 따라 잘게 분리되어 있습니다. (설계 관점 분리 적용)

- **`main.py`** : 프로그램이 가장 먼저 실행되는 곳입니다. 사용자에게 메뉴를 보여주고, 입력을 넘겨주는 진입점 역할을 담당합니다.
- **`quiz.py`** : 퀴즈 한 문제의 데이터를 담는 `Quiz` 클래스가 정의되어 있습니다. JSON 형식으로의 변환과 복원을 책임집니다.
- **`game.py`** : 퀴즈 로드, 풀기, 추가, 삭제, 채점 등 퀴즈 게임의 거의 모든 실제 두뇌 동작을 통제하는 `QuizGame` 클래스가 있습니다.
- **`data.py`** : 최초 실행 시 퀴즈가 하나도 없는 빈 상태일 경우, 제공하기 위한 영화 주제의 초기 기본 문제 모델들을 제공해 줍니다.
- **`.gitignore`** : Git이 버전 관리를 할 때 불필요한 캐시 파일이나 폴더 내용물들을 눈감아주도록 규칙들이 적혀있습니다.

## 🌿 8. Git 워크플로우 통제 및 복제 실습 내용

본 프로젝트는 그저 코드를 짜는 것을 넘어 실무 팀 단위 코드 작성처럼 **Git** 이라는 형상 관리 도구를 철저하게 곁에 두고 관리되었습니다.

### 필수 기초 명령어 (7종) 사용 확인!
저장소는 최초로 `git init`에 의해 로컬에 뿌리를 내렸고, `git add`와 `git commit`을 반복적으로 수행하여 약 15건 이상의 기능별 작업(보너스, 퀴즈, 메뉴 추가 등) 이력을 분리해 남겼습니다. 이후 `git push`를 통해 이곳 GitHub까지 데이터가 배달되었습니다.

특히나 퀴즈 풀기 기능을 안전하게 실험하기 위해, `git checkout -b feature/play-quiz` 명령을 써서 메인 줄기가 아닌 임시 공간(브랜치)에서 개발하다가 무사히 `merge`된 특별한 경험 기록도 존재합니다!

### 두근두근 Clone & Pull 복제 실습 체험
실무에서의 동료와의 협업을 가상해 로컬 디렉토리상에서 아래의 시나리오를 통과했습니다.
1. 이 완성된 전체 저장소를 `git clone` 명령어로 동일 환경에 `1-B-practice`라는 제2의 폴더로 통째로 복제해냈습니다.
2. 복제된 가상의 동료 폴더(`1-B-practice`) 안에서 README 파일에 테스트 텍스트 수정 작업을 거치고, `push`를 진행했습니다.
3. 이를 다시 본래 주인의 폴더(`1-B`)로 돌아와 `git pull` 명령어로 당겨 받아들였습니다. **그 결과 수동 타이핑 없이 변경 내역이 완벽히 다운로드 및 반영되는 과정을 지켜보고 터득해냈습니다.**

## 💡 9. 아키텍처 및 핵심 개념 Q&A (평가 기준 검증)

본 프로젝트를 구현하며 다듬어진 필수 파이썬/Git 개념들과 설계 결정들에 대한 해답입니다.

**Q. `Quiz`와 `QuizGame` 두 가지 클래스들의 책임을 어떻게 나누었나요? (평가기준 8)**  
A. `Quiz` 클래스는 **'데이터 모델'**역할을 하도록 퀴즈의 문항 내용(문제, 선택지, 정답, 힌트) 보관과 정답 채점 등 개별 객체 단위의 책임을 쥐고 있습니다. 반면 `QuizGame` 클래스는 이 퀴즈들을 조종하여 화면에 출력하고 플레이어의 입력을 제어하며, 점수나 게임의 히스토리 상태를 관리하고 파일을 저장/불러오는 **'게임 통제 컨트롤러'** 역할을 수행합니다.

**Q. "입력 처리(검증)", "게임 진행", "데이터 저장/불러오기" 로직 분리 기준은? (평가기준 9)**  
A. 
- **입력 처리:** 철저히 사용자의 입력을 받는 부분(`main()` 메뉴의 `get_menu_choice()` 또는 게임 진행 중 `input()`)에 배치하여, 정상적인 값이 들어올 때까지 무한 반복(`while True` + `try/except`)으로 방어했습니다. 핵심 비즈니스 로직에 나쁜 데이터가 전파되는 것을 최전선에서 막기 위함입니다.
- **게임 진행:** `QuizGame` 안의 `play()`, `add_quiz()` 등으로 묶어, 철저하게 게임의 비즈니스 룰(점수 계산, 정답률 로직)에 집중하도록 구성했습니다.
- **데이터 처리:** `save_state()`, `load_state()` 함수로 완전히 격리했습니다. 게임 로직은 데이터가 어디서 어떻게 오는지 알 필요 없이, 단순히 리스트(`self.quizzes`)에만 접근하도록 의존성을 줄였습니다.

**Q. `state.json` 읽기/쓰기 흐름이 어느 순서로 발생하나요? (평가기준 10)**  
A. 프로그램이 실행되어 `QuizGame` 인스턴스가 생성되는 즉시 `__init__`에서 맨 처음 `load_state()`가 호출되어 데이터를 읽습니다. 역으로, 게임 중 퀴즈를 풀고 점수가 갱신될 때마다, 퀴즈를 삭제하거나 추가할 때마다, 그리고 사용자가 메뉴에서 `(6) 종료`를 누를 때마다 곧바로 `save_state()`가 실행되며 실시간으로 변경 사항을 파일에 기록합니다.

**Q. `Ctrl+C` 또는 `EOF` 상황에서 "안전 종료"를 위한 예외 처리 방식은? (평가기준 11)**  
A. 프로그램의 모든 `input()`과 메인 루프를 `try / except (KeyboardInterrupt, EOFError):` 블록으로 감쌌습니다. 에러가 잡히면 비정상 종료(Traceback)를 막고 "게임 진행이 중단되었습니다"라는 안내와 함께 그 시점까지의 상태를 `save_state()`로 강제 저장한 후 `sys.exit(0)` 혹은 `return`을 호출하여 우아하게(graceful) 안전 종료시킵니다.

**Q. 커밋 단위와 커밋 메시지 규칙은 어떻게 정했나요? (평가기준 12)**  
A. 커밋은 기능 단위(예: 'Feat: 퀴즈 추가 기능', 'Refactor: 책임 분리')로 세밀하게 나누었습니다. 메시지 헤더는 `유형: 제목` 형태로 통일했고, `Feat`(새로운 기능), `Fix`(오류 수정), `Docs`(문서 수정), `Refactor`(기능 변화 없이 구조 개선) 접두사를 사용하여 다른 팀원이 수정 의도를 명확하게 알 수 있도록 했습니다.

**Q. 클래스를 사용한 이유와, 함수만 썼을 때의 차이점은? (평가기준 13)**  
A. 클래스를 사용하면 데이터(상태)와 그 데이터를 조작하는 메서드(행동)를 하나로 묶어 독립적인 캡슐(인스턴스)로 관리할 수 있습니다. 만약 함수와 전역 변수들로만 짰다면, 향후 퀴즈 배열이 복잡해질수록 변수 오염이 쉽고 코드를 읽기 어려우며 추적이 곤란해집니다.

**Q. JSON 파일로 데이터를 저장하는 이유와 특징은? (평가기준 14)**  
A. JSON (JavaScript Object Notation)은 텍스트 형태이므로 **파이썬 뿐만 아니라 거의 모든 언어에서 아주 쉽게 읽고 쓰고 호환**될 수 있습니다. 특히 파이썬의 리스트나 딕셔너리 자료형과 1:1로 정확히 직렬화/역직렬화 매칭이 되기 때문에 가장 이상적입니다.

**Q. 파일 입출력에서 `try/except`가 필요한 이유는? (평가기준 15)**  
A. 존재하지 않는 파일을 읽으려 하거나, 파일이 손상되었거나, 운영체제 상에서 권한(퍼미션) 오류가 발생하는 등 파이썬 코드 외적인 I/O 오류는 런타임에 불규칙적으로 일어납니다. 이를 예외 처리하지 않으면 프로그램 전체가 그 즉시 치명적으로 종료되기 때문입니다.

**Q. 브랜치를 분리해 작업한 이유와 병합(merge)의 의미는? (평가기준 16)**  
A. 브랜치를 분리(`feature/play-quiz` 등)하면, 현재 잘 돌아가는 원본 중심선(`main`) 코드는 망가뜨리지 않고 복사본에서 안전하게 실험 및 개발할 수 있습니다. 모든 테스트가 끝나 코드가 증명됐을 때, 그 변경사항의 결과만 `main`에 합치는 작업이 바로 병합(`merge`)입니다.

**Q. `state.json` 구조를 현재처럼 배열 및 딕셔너리 구조(`quizzes`/`best_score`/`history`)로 묶어놓은 이유는? (평가기준 17)**  
A. 최상위를 하나의 딕셔너리(Object) `{}`로 닫고 각 기능(퀴즈 목록, 기본 정보)을 키로 분리했습니다. 이렇게 해야 향후 `settings`(게임 설정값)이나 `user_info` 등 새로운 형태의 데이터 덩어리가 추가되더라도, 기존 구조를 전혀 깨뜨리지 않고 루트 객체의 새 필드로 유연하게 집어넣을 수 있는 **확장성**을 갖게 됩니다.

**Q. 퀴즈 데이터가 1000개 이상으로 늘어난다면 JSON 저장 방식에 어떤 한계가? (평가기준 18)**  
A. 현재 방식은 파일을 매번 `load` 할 때 전체 파일 데이터를 서버 메모리에 모두 띄우고(디코딩), `save` 할 때도 메모리에 있는 1000개의 상태를 통째로 모아서 다시 처음부터 텍스트(인코딩)로 덮어씁니다. 데이터가 막대해지면 입출력(Disk I/O) 병목현상과 메모리 과부하(OOM)가 오며 게임 동작이 크게 지연될 수 있습니다. (이때는 RDBMS나 NoSQL 같은 DB가 필요합니다.)

**Q. `state.json`이 손상되어 파싱에 실패한다면, 어떤 대응 방안이? (평가기준 19)**  
A. `try/except json.JSONDecodeError`를 통해 파싱 오류를 붙잡고, 에러 로그 출력 후 본 게임 내 기본으로 탑재된 `data.py`(초기 5개 문제 템플릿)의 데이터를 메모리에 복구시켜 게임 자체는 지장 없이 실행하도록 안전망(Fallback)이 구축되어 있습니다.

**Q. "정답 채점 방식"이나 "퀴즈 구조"가 변경된다면 제일 먼저 어딜 고쳐야 하나요? (평가기준 20)**  
A. 책임을 잘게 분리했기 때문에, 퀴즈 자체가 품고 있는 정보와 관련된 것(예: 주관식으로 변경)이라면 `quiz.py`의 `Quiz` 클래스 내부 속성과 메서드만 뜯어고치면 됩니다. 만약 채점의 점수 비율이나 로직(예: 3문제 연속 정답 시 콤보 점수 적용)이 변경된다면, 게임 흐름을 담당하는 `game.py` 안의 `QuizGame.play()` 메서드 쪽을 중점적으로 수정하면 됩니다.
